# Backend Design Patterns

이 문서는 EgoFlow backend의 현재 서버 구조와 설계 패턴을 큰 그림에서 정리한다.
세부 구현 스타일보다 파일 역할, 호출 방향, DTO와 mapper의 위치를 기준으로 본다.

## 1. 기본 계층 흐름

Backend API는 다음 흐름을 기본 구조로 둔다.

```text
route -> middleware -> service -> lib -> repository -> prisma -> DB
```

각 계층은 아래 책임을 가진다.

| 계층 | 위치 | 책임 |
| --- | --- | --- |
| route | `backend/src/routes/*` | HTTP method/path, middleware 연결, request parsing 결과 전달, response status/cookie/header 처리 |
| middleware | `backend/src/middleware/*` | 인증, 권한, repository 상태 확인, validation, request context 주입 |
| service | `backend/src/services/*` | route use-case orchestration, 여러 repository/lib 호출 조합, 응답 조립 |
| lib | `backend/src/lib/*` | route에서 직접 호출하지 않는 도메인 component, 재사용 로직, 외부 시스템/Redis/storage helper |
| repository | `backend/src/repositories/*` | Prisma query를 감싼 table 단위 persistence 계층 |
| prisma | `backend/src/lib/infra/prisma.ts` | Prisma client, transaction helper |
| DB | PostgreSQL | 실제 영속 데이터 저장소 |

## 2. Route와 Service

API route 파일은 자기 도메인의 service 하나를 직접 호출한다.

예시는 다음과 같다.

| route | service |
| --- | --- |
| `routes/auth.routes.ts` | `services/auth.service.ts` |
| `routes/admin.routes.ts` | `services/admin.service.ts` |
| `routes/repositories.routes.ts` | `services/repository.service.ts` |
| `routes/videos.route.ts` | `services/videos.service.ts` |
| `routes/streams.routes.ts` | `services/stream.service.ts` |
| `routes/live-streams.routes.ts` | `services/live-streams.service.ts` |
| `routes/http-streams.routes.ts` | `services/http-stream.service.ts` |
| `routes/hooks.routes.ts` | `services/hooks.service.ts` |
| `routes/recordings.routes.ts` | `services/recordings.service.ts` |

route는 HTTP 표면을 다룬다.

- URL path와 HTTP method를 정의한다.
- `validate(...)`, auth middleware, repository middleware를 연결한다.
- `getAuthUser(req)`, `getRepositoryAccessContext(req)`처럼 middleware가 만든 request context를 읽는다.
- service 호출 결과를 적절한 status code로 내려준다.
- dashboard session cookie처럼 HTTP response 객체가 필요한 처리는 route에서 수행한다.

route는 다른 도메인의 service를 직접 호출하지 않는다.
다른 도메인 데이터가 필요하면 service가 lib 또는 repository를 통해 orchestration한다.

## 3. Middleware 적용 방식

middleware는 route 진입 전에 공통 검증과 context 생성을 담당한다.

주요 middleware는 다음과 같다.

| middleware | 역할 |
| --- | --- |
| `auth.middleware.ts` | dashboard session, app JWT, Python token 인증 |
| `role.middleware.ts` | system admin 등 role 기반 접근 제어 |
| `repository.middleware.ts` | repository access/status 확인 후 request context에 저장 |
| `validate.middleware.ts` | Zod schema 기반 body/query/params 검증 |
| `http-stream.middleware.ts` | raw chunk upload header/body 파싱 |
| `signed-file-access.middleware.ts` | signed `/files/*` 접근 검증 |
| `error.middleware.ts` | AppError/Zod/Prisma/unknown error를 HTTP response로 변환 |

repository 관련 route는 보통 아래 흐름을 가진다.

```text
auth middleware
-> repoAccess / repoStatus middleware
-> validate middleware
-> route handler
-> service
```

`repoAccess`는 `repositoryAccessService`를 사용해 action별 최소 role과 repository visibility를 해석한다.
성공하면 `RepositoryAccessContext`가 request context에 저장되고, service는 이 context를 신뢰하고 사용한다.

## 4. Service의 역할

service는 route use-case의 중심 orchestration 계층이다.

service가 담당하는 일:

- 검증된 request input과 request context를 받아 use-case를 실행한다.
- 여러 table repository 호출을 조합한다.
- 필요한 domain component를 `lib/*`에서 호출한다.
- mapper를 사용해 response DTO를 조립한다.
- 비즈니스 예외를 `BadRequest`, `NotFound`, `Conflict`, `Forbidden` 같은 AppError로 표현한다.

service가 하지 않는 일:

- Prisma client를 직접 호출하지 않는다.
- 다른 route domain service를 직접 호출하지 않는다.
- route response DTO 타입을 service 파일 내부에 새로 정의하지 않는다.
- 재사용 가능한 helper 구현을 service 내부에 오래 두지 않는다.

예를 들어 HTTP upload timeout 복구/reconcile 로직은 route service가 아니라
`lib/streaming/http-upload-session.ts`에 component로 분리되어 있고,
`stream.service.ts`는 reconcile loop에서 그 lib 함수를 호출한다.

## 5. Lib Component

`lib/*`는 route에서 직접 호출되지 않는 재사용 component를 둔다.
DB state가 필요하면 Prisma를 직접 import하지 않고 repository를 호출한다.

대표적인 lib 영역은 다음과 같다.

| 위치 | 역할 |
| --- | --- |
| `lib/auth/*` | access token, dashboard session, password, Python token helper |
| `lib/repositories/*` | repository access policy, role comparison, contributor refresh |
| `lib/streaming/*` | stream path, ownership ticket, live cache, recording session lifecycle, HTTP upload session |
| `lib/processing/*` | BullMQ queue, finalize progress helper |
| `lib/storage/*` | file path, signed URL, storage directory helper |
| `lib/http/*` | async handler, request context helper |
| `lib/infra/*` | Prisma, Redis, BullMQ infra client |

lib는 service보다 더 작은 domain component다.
service는 endpoint use-case를 조립하고, lib는 재사용 가능한 세부 도메인 기능을 제공한다.

## 6. Repository 구조

repository 파일은 DB table 하나에 대응하는 persistence 계층이다.

예시는 다음과 같다.

| repository | 주요 table |
| --- | --- |
| `repositories.repository.ts` | `repositories` |
| `repo-member.repository.ts` | `repo_members` |
| `videos.repository.ts` | `videos` |
| `recording-session.repository.ts` | `recording_sessions` |
| `recording-segment.repository.ts` | `recording_segments` |
| `user.repository.ts` | `users` |
| `python-token.repository.ts` | `python_tokens` |
| `setting.repository.ts` | `settings` |
| `video-semantic-metadata.repository.ts` | `video_semantic_metadata` |

repository는 Prisma query를 감싸고, select/include shape와 persistence row 타입을 관리한다.
여러 table을 함께 다루는 use-case는 repository 하나에 넣지 않고 service 또는 lib에서 orchestration한다.

## 7. Schema와 Request DTO

request validation은 `backend/src/schemas/*`의 Zod schema가 담당한다.
schema 파일은 runtime validation schema를 export하고, DTO type alias는 export하지 않는다.

request DTO type은 도메인별로 아래 위치에 둔다.

```text
backend/src/types/{domain}/request.ts
```

예시는 다음과 같다.

| domain | request DTO |
| --- | --- |
| auth | `types/auth/request.ts` |
| admin | `types/admin/request.ts` |
| repository | `types/repository/request.ts` |
| videos | `types/videos/request.ts` |
| stream | `types/stream/request.ts` |

request DTO는 Zod schema에서 `z.infer<typeof schema>`로 만든다.
route와 service는 schema 파일에서 type을 가져오지 않고 `types/{domain}/request.ts`에서 가져온다.

## 8. Response DTO와 Mapper

route response DTO type은 도메인별로 아래 위치에 둔다.

```text
backend/src/types/{domain}/response.ts
```

예시는 다음과 같다.

| domain | response DTO |
| --- | --- |
| auth | `types/auth/response.ts` |
| admin | `types/admin/response.ts` |
| repository | `types/repository/response.ts` |
| videos | `types/videos/response.ts` |
| stream | `types/stream/response.ts` |
| processing | `types/processing/response.ts` |

DB row 또는 internal model을 response DTO로 바꾸는 함수는 mapper에 둔다.

```text
backend/src/mappers/*.mapper.ts
```

현재 mapper 예시는 다음과 같다.

| mapper | 역할 |
| --- | --- |
| `user.mapper.ts` | Prisma user role/user record를 app auth/user response로 변환 |
| `admin.mapper.ts` | admin user/settings response 변환 |
| `repository.mapper.ts` | repository record/summary response 변환 |
| `video.mapper.ts` | repository video/contributor/status response 변환 |

service는 response object의 세부 shape를 직접 정의하기보다 mapper와 response DTO type을 사용한다.

## 9. Domain Type Entry Point

도메인별 type 폴더는 필요에 따라 세 파일로 나뉜다.

```text
types/{domain}/model.ts
types/{domain}/request.ts
types/{domain}/response.ts
types/{domain}/index.ts
```

- `model.ts`: 내부 domain model 또는 context 타입
- `request.ts`: request DTO type
- `response.ts`: response DTO type
- `index.ts`: 해당 domain type의 public entrypoint

예를 들어 repository domain은 다음처럼 구성된다.

```text
types/repository/model.ts
types/repository/request.ts
types/repository/response.ts
types/repository/index.ts
```

일반 import는 `../types/repository`처럼 index를 통해 할 수 있고,
request/response 구분이 중요한 곳은 `../types/repository/request`처럼 직접 가져온다.

## 10. Constants

재사용되는 상수는 `backend/src/constants/*` 아래에 둔다.

예시는 다음과 같다.

| 위치 | 역할 |
| --- | --- |
| `constants/auth/auth-constants.ts` | auth credential kind, token prefix/TTL |
| `constants/config/config-constants.ts` | fixed port/config value |
| `constants/stream/stream-constants.ts` | stream Redis key, TTL, timeout |
| `constants/stream/stream-ownership-constants.ts` | publish ticket TTL |
| `constants/storage/*` | storage/signed file URL 상수 |
| `constants/processing/processing-constants.ts` | recording finalize step 목록 |

service나 lib 안에 반복 사용되는 magic value를 새로 만들 때는 먼저 constants 위치를 고려한다.

## 11. 현재 구조의 의도

현재 backend 구조의 핵심 의도는 다음과 같다.

- route는 HTTP 표면만 다룬다.
- middleware는 공통 검증과 request context 생성을 담당한다.
- service는 단일 route domain의 use-case orchestration을 담당한다.
- service 내부의 재사용 가능한 구현은 lib component로 분리한다.
- repository는 table 단위 persistence만 담당한다.
- request DTO와 response DTO는 `types/{domain}` 아래에서 통합 관리한다.
- response 변환은 mapper에 모아 service 파일의 책임을 줄인다.

이 구조를 유지하면 endpoint가 늘어나도 route/service/repository의 역할이 섞이지 않고,
DTO 변경이나 response shape 변경도 도메인 type과 mapper 중심으로 추적할 수 있다.
