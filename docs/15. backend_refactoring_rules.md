# Backend Refactoring Rules

## 기본 호출 흐름

Backend API는 기본적으로 다음 계층 순서를 따른다.

`route -> middleware -> service -> lib -> repository -> prisma`

- `route`: HTTP method/path, request parsing 결과, response status/cookie/header만 다룬다.
- `middleware`: 인증, 권한, repository 상태, validation처럼 route 진입 전 공통 검증을 담당한다.
- `service`: route use-case의 orchestration을 담당한다. 일반적으로 Prisma를 직접 호출하지 않는다.
- `lib`: route에서 직접 호출되지 않는 재사용 구현체를 둔다. 저장소 접근이 필요하면 repository를 통해 호출한다.
- `repository`: Prisma query를 감싸는 persistence 계층이다. 일반적인 DB 접근은 이 계층에서만 수행한다.

## Service 이름 규칙

특정 route에서 직접 호출하는 service는 route 도메인과 이름을 맞춘다.

- `auth.routes.ts`는 `auth.service.ts`를 호출한다.
- `repository.routes.ts`는 `repository.service.ts`를 호출한다.
- route handler에서 호출하는 service method는 endpoint use-case 이름이 드러나도록 둔다.

## Helper 위치 규칙

route에서 직접 호출되지 않는 helper는 service로 두지 않고 `lib` 아래에 둔다.

- session token 생성/검증, password hash, signed token 처리 같은 구현체는 `lib/auth/*`에 둔다.
- 파일 경로 계산, MIME 처리, crypto helper처럼 도메인 use-case가 아닌 구현체도 `lib/*`에 둔다.
- `lib`가 DB state를 읽거나 써야 하면 직접 Prisma를 import하지 않고 repository를 호출한다.

## 예외

HTTP cookie 설정/삭제처럼 `Response` 객체가 필요한 코드는 route 또는 HTTP 전용 middleware/lib에서 처리할 수 있다.
worker, bootstrap, migration, test helper처럼 API route 흐름 밖에 있는 코드는 예외가 될 수 있지만, 예외는 파일 역할이 명확해야 한다.
