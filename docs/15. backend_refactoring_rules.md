# Backend Refactoring Rules

## 기본 호출 흐름

Backend API는 기본적으로 다음 계층 순서를 따른다.

`route -> middleware -> service -> lib -> repository -> prisma`

- `route`: HTTP method/path, request parsing 결과, response status/cookie/header만 다룬다.
- `middleware`: 인증, 권한, repository 상태, validation처럼 route 진입 전 공통 검증을 담당한다.
- `service`: route use-case의 orchestration을 담당한다. 일반적으로 Prisma를 직접 호출하지 않는다.
- `lib`: route에서 직접 호출되지 않는 재사용 구현체를 둔다. 저장소 접근이 필요하면 repository를 통해 호출한다.
- `repository`: Prisma query를 감싸는 persistence 계층이다. 일반적인 DB 접근은 이 계층에서만 수행한다.

## Service 이름 규칙

특정 route에서 직접 호출하는 service는 route 도메인과 이름을 맞춘다.

- `auth.routes.ts`는 `auth.service.ts`를 호출한다.
- `repositories.routes.ts`는 `repositories.service.ts`를 호출한다.
- route handler에서 호출하는 service method는 endpoint use-case 이름이 드러나도록 둔다.

## Helper 위치 규칙

route에서 직접 호출되지 않는 helper는 service로 두지 않고 `lib` 아래에 둔다.

- session token 생성/검증, password hash, signed token 처리 같은 구현체는 `lib/auth/*`에 둔다.
- 파일 경로 계산, MIME 처리, crypto helper처럼 도메인 use-case가 아닌 구현체도 `lib/*`에 둔다.
- `lib`가 DB state를 읽거나 써야 하면 직접 Prisma를 import하지 않고 repository를 호출한다.

## Constant / API Format 규칙

상수와 API 입출력 포맷은 구현 파일 안에 흩어두지 않는다.

- 재사용되는 magic value, TTL, key prefix, fixed port, enum-like string은 `constants/*` 아래에 둔다.
- request validation은 `schemas/*`의 Zod schema가 담당하고, schema 파일은 DTO type alias를 export하지 않는다.
- request DTO alias는 `types/{domain}/request.ts`에 둔다.
- route response DTO와 external API payload format은 `types/{domain}/response.ts`에 둔다.
- 도메인 내부 모델 타입은 필요할 때 `types/{domain}/model.ts`에 두고, `types/{domain}/index.ts`가 도메인 타입의 public entrypoint가 된다.
- DB row/persistence 전용 타입은 table repository 내부에 둘 수 있지만, route response 타입으로 재사용하지 않는다.
- service는 DTO shape를 새로 정의하지 않고, `types/*` 타입과 mapper/helper를 사용해 response를 조립한다.

## Route-Service 대응 규칙

API route 파일은 동일 도메인의 service 파일 하나만 직접 호출한다.

- `routes/auth.routes.ts` → `services/auth.service.ts`
- `routes/repositories.routes.ts` → `services/repositories.service.ts`
- `routes/videos.route.ts` → `services/videos.service.ts`

route handler에서 다른 route 도메인의 service를 직접 호출하지 않는다. 다른 도메인 데이터가 필요하면 route service가 use-case를 orchestration하고, 필요한 DB access는 table repository를 통해 수행한다.

## Repository 대응 규칙

Repository 파일은 DB table 하나에 대응한다.

- `repositories` table 접근은 repository table 전용 repository에 둔다.
- `repo_members` table 접근은 repo member table 전용 repository에 둔다.
- `videos` table 접근은 video table 전용 repository에 둔다.
- `recording_sessions`, `recording_segments`, `users`, `python_tokens`, `settings`, `video_semantic_metadata`도 같은 규칙을 따른다.

하나의 repository 파일에서 여러 Prisma model을 직접 호출하지 않는다. 여러 table을 함께 다루는 use-case는 service 또는 `lib`가 orchestration하고, 각 table access는 해당 table repository 메서드를 호출한다.

## Prisma model naming 규칙

Prisma model 이름은 DB table 이름에서 `_`를 제거하고 PascalCase로 변환한 이름을 사용한다.
DB table 이름은 PostgreSQL 관례대로 snake_case를 유지하고, Prisma model에는 `@@map("table_name")`을 둔다.

예시는 다음과 같다.

| DB table | Prisma model | Prisma client accessor |
| --- | --- | --- |
| `users` | `Users` | `prisma.users` |
| `python_tokens` | `PythonTokens` | `prisma.pythonTokens` |
| `repositories` | `Repositories` | `prisma.repositories` |
| `repo_members` | `RepoMembers` | `prisma.repoMembers` |
| `recording_sessions` | `RecordingSessions` | `prisma.recordingSessions` |
| `recording_segments` | `RecordingSegments` | `prisma.recordingSegments` |
| `videos` | `Videos` | `prisma.videos` |
| `video_semantic_metadata` | `VideoSemanticMetadata` | `prisma.videoSemanticMetadata` |

Prisma field 이름은 application code에서 쓰기 좋은 camelCase를 사용하고, DB column이 snake_case이면 `@map("column_name")`으로 연결한다.
새 table을 추가할 때도 DB table/column 이름은 snake_case, Prisma model/field 이름은 PascalCase/camelCase로 유지한다.

## 비 Route Service 위치 규칙

`services` 디렉터리에는 route와 1:1로 대응되는 service만 둔다.

route에 직접 대응하지 않는 구현체는 `lib` 하위로 이동한다.

- 인증 helper: `lib/auth/*`
- streaming/session ownership/reconcile helper: `lib/streaming/*`
- processing/finalize helper: `lib/processing/*`
- repository access/contributor helper: `lib/repositories/*`

## 예외

HTTP cookie 설정/삭제처럼 `Response` 객체가 필요한 코드는 route 또는 HTTP 전용 middleware/lib에서 처리할 수 있다.
worker, bootstrap, migration, test helper처럼 API route 흐름 밖에 있는 코드는 예외가 될 수 있지만, 예외는 파일 역할이 명확해야 한다.
