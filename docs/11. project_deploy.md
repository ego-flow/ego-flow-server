# EgoFlow Server Deploy

이 문서는 현재 `ego-flow-server`의 실행 및 서버 재기동 방식을 정리한 문서다. 로컬과 원격 서버 모두 같은 `compose.yml`과 `./scripts/run.sh`를 사용한다.

## 1. 기본 원칙

- 스택 실행 진입점은 `ego-flow-server/scripts/run.sh up` 하나다.
- 로컬 머신과 원격 Linux 서버가 같은 Docker Compose 자산을 사용한다.
- 운영 서버에서 코드 최신화와 스택 재기동은 parent repo 루트의 `./scripts/server-up.sh`가 담당한다.
- 별도 prod compose, image registry pull, remote config upload 단계는 없다.

## 2. 로컬 실행

```bash
cd ego-flow-server
cp config.json.example config.json
cp .env.example .env
./scripts/run.sh doctor
./scripts/run.sh up
```

`./scripts/run.sh up`은 다음을 수행한다.

1. Docker / Compose / config 파일 존재 여부 확인
2. `config.json`의 `TARGET_DIRECTORY`를 현재 host data root로 해석하고 `.run/target-directory` 기준 이전 root migration 여부 확인
3. `docker compose -f compose.yml up -d --build --remove-orphans`
4. `postgres`, `redis`, `backend`, `dashboard`, `proxy` health check 대기
5. `worker`, `mediamtx` running 상태 대기

시작 후 확인 주소:

- Backend health: `http://localhost:{PUBLIC_HTTP_PORT}/api/v1/health`
- Swagger UI: `http://localhost:{PUBLIC_HTTP_PORT}/api-docs`
- Dashboard: `http://localhost:{PUBLIC_HTTP_PORT}`
- RTMP ingest: `rtmp://localhost:1935/live`
- RTMPS ingest: `rtmps://localhost:1936/live`
- HLS output: `http://localhost:{PUBLIC_HTTP_PORT}/hls`

## 3. 원격 서버 실행

원격 서버도 같은 방식으로 동작한다. 차이는 저장소가 parent repo 기준이라는 점뿐이다.

### 3.1 최초 준비

```bash
git clone <repo-url>
cd ego-flow/ego-flow-server
./scripts/setup-server-config.sh
```

스크립트는 `config.json`과 `.env`에 들어갈 값을 순서대로 물어보고, 입력된 값으로 두 파일을 생성한다. 기존 `.env` 또는 `config.json`이 있으면 입력 완료 후 덮어쓴다.

서버에 맞게 주로 조정하는 값:

- `config.json`
  - `TARGET_DIRECTORY`
  - `PUBLIC_HTTP_PORT`
- `.env`
  - `JWT_SECRET`
  - `ADMIN_DEFAULT_PASSWORD`
  - 필요 시 `RTMPS_ENCRYPTION_MODE`, `RTMPS_CERT_PATH`, `RTMPS_KEY_PATH`

다른 장치가 이 서버에 붙는 경우에도 RTMP/WHIP publish URL은 app이 backend origin과 publish-ticket 응답의 `stream_path`, `publish_ticket`으로 직접 조립한다.
HLS playback은 별도 public base URL을 두지 않고 `/hls/...` relative path를 반환하므로, frontend는 이 경로를 그대로 사용하고 브라우저가 현재 dashboard origin 기준으로 요청한다.

`TARGET_DIRECTORY` 아래에는 아래 구조가 만들어진다.

```text
{TARGET_DIRECTORY}/
├── postgres/
├── redis/
├── raw/
└── datasets/
```

### 3.2 표준 재기동 경로

```bash
cd ego-flow
./scripts/server-up.sh
```

`server-up.sh`는 다음을 수행한다.

1. 기존 stack이 실행 중이면 `ego-flow-server/scripts/run.sh down`
2. parent repo `git pull --ff-only`
3. `git submodule sync --recursive`
4. `git submodule update --init --recursive`
5. `ego-flow-server/scripts/run.sh up`

최종 readiness 판단은 `run.sh up` 내부 health wait를 사용한다.

## 4. 운영 명령

```bash
cd ego-flow-server
./scripts/run.sh ps
./scripts/run.sh logs backend
./scripts/run.sh logs mediamtx
./scripts/run.sh down
```

명령 의미:

- `ps`: 현재 서비스 상태 확인
- `logs [service]`: 전체 또는 특정 서비스 로그 추적
- `down`: 컨테이너 종료 및 제거
- `up`: 이미지 build 포함 전체 스택 재기동

## 5. 운영상 기억할 점

- 운영 서버의 persistent data는 모두 `TARGET_DIRECTORY/{postgres,redis,raw,datasets}` 아래에 저장된다.
- `./scripts/run.sh up`은 `.run/target-directory`에 기록된 이전 host data root와 현재 `TARGET_DIRECTORY`를 비교해서 필요 시 host data root migration을 먼저 수행한다.
- backend는 부팅 시 `settings.target_directory`와 현재 `{TARGET_DIRECTORY}/datasets`를 비교해 generated dataset path를 재작성한다.
- `./scripts/run.sh reset`은 `TARGET_DIRECTORY` 전체와 `.run/target-directory` state를 제거하므로 일반 서버 운영 절차에 포함하지 않는다.
- RTMPS를 실제로 켤 때는 cert/key 파일 준비와 `RTMPS_ENCRYPTION_MODE` 설정이 필요하다. RTMPS publish URL도 app에서 고정 포트 `1936` 기준으로 조립한다.
- 앱은 backend host 설정만 바꾸면 같은 API/streaming 계약으로 서버에 접속한다.
