# EgoFlow Server Authentication

이 문서는 현재 `ego-flow-server`의 인증과 인가 구조를 정리한 문서다. 현재 구현은 접근 매체별 인증 창구를 분리한다.

## 1. 인증 창구

| 접근 매체 | credential | 주요 용도 |
| --- | --- | --- |
| Dashboard | `egoflow_session` HttpOnly cookie | web dashboard API, admin, profile, video browsing |
| Android app | `Authorization: Bearer <JWT>` | stream register, publish-ticket, heartbeat, stop |
| Python package | `Authorization: Bearer <ef_...>` | dataset manifest, video download, thumbnail download |
| HLS playback | `egoflow_session` cookie 또는 `Authorization: Bearer <JWT|ef_...>` | Caddy `forward_auth`를 통한 live HLS playback auth |
| Signed file URL | `/files/*?signature=<JWT>` | generated file bytes after prior API authorization |

route는 `requireDashboardSession`, `requireAppJwt`, `requirePythonToken`, `requireDashboardOrPython`, `requireDashboardOrApp`, `requireDashboardOrAppOrPython`처럼 명시적인 인증 component를 사용한다.

## 2. Dashboard Session

Dashboard는 bearer JWT를 브라우저 저장소에 보관하지 않는다.

- 로그인 endpoint: `POST /api/v1/auth/dashboard/login`
- session 확인 endpoint: `GET /api/v1/auth/dashboard/session`
- logout endpoint: `POST /api/v1/auth/dashboard/logout`
- cookie 이름: `egoflow_session`
- cookie 속성: `HttpOnly`, `SameSite=Lax`, `Path=/`, public 요청이 실제 HTTPS일 때만 `Secure`
- session 저장소: Redis
- cookie에는 opaque random session token만 들어가고, Redis key에는 SHA-256 hash만 사용한다.
- Redis key: `dashboard:session:{sha256(raw_session_token)}`
- `remember_me=false` session은 browser session cookie이고 서버 만료는 12시간이다.
- `remember_me=true` session은 persistent cookie이고 서버 만료는 30일이다.
- `lastUsedAt`은 요청마다 쓰지 않고 5분 간격으로만 갱신한다.
- dashboard login page keyboard flow는 login form 내부 어디서든 `Enter` -> current login submit으로 동작한다.

Dashboard frontend는 `axios`를 `withCredentials: true`로 호출한다. page refresh 시 `/auth/dashboard/session`으로 현재 session을 복구한다.
즉 reverse proxy 뒤 HTTP 개발/운영 환경에서도 cookie가 저장되고, HTTPS 배포에서는 `X-Forwarded-Proto: https` 또는 `req.secure` 기준으로 자동으로 `Secure`가 적용된다.

## 3. App JWT

Android app은 기존 bearer JWT 방식을 유지한다.

- legacy endpoint: `POST /api/v1/auth/login`
- 명시적 app endpoint: `POST /api/v1/auth/app/login`
- 응답: `{ token, user }`
- 서명 방식: HS256
- refresh token: 없음
- 토큰 갱신: app JWT 경로에서만 필요 시 `X-Refreshed-Token` 응답 헤더로 새 token 반환

app JWT는 streaming 관련 API에만 사용한다.

- `GET /api/v1/repositories/mine`
- `GET /api/v1/repositories/resolve`
- `POST /api/v1/streams/register`
- `POST /api/v1/streams/:recordingSessionId/publish-ticket`
- `POST /api/v1/streams/:recordingSessionId/connections/:connectionId/heartbeat`
- `POST /api/v1/recordings/:recordingSessionId/stop`
- `GET /api/v1/recordings/:recordingSessionId`

## 4. Python Token

Python package와 장기 실행 환경은 Python token을 사용한다.

- 형식: `ef_` + 40자 hex
- 발급 endpoint: `POST /api/v1/auth/tokens`
- 조회 endpoint: `GET /api/v1/auth/tokens`
- revoke endpoint: `DELETE /api/v1/auth/tokens/:tokenId`
- 저장 방식: `api_tokens` 테이블에 SHA-256 hash만 저장
- 사용자당 active token은 정확히 1개
- 재발급 시 기존 row를 hard delete하고 새 row만 남긴다
- 원본 token은 발급 직후 한 번만 응답에 포함되고 이후 재조회할 수 없다

token 발급/조회/revoke 자체는 dashboard session이 필요하다. 발급된 Python token은 Python dataset API에 사용한다.

- `GET /api/v1/repositories/:repoId/manifest`
- `GET /api/v1/repositories/resolve`
- `GET/HEAD /api/v1/repositories/:repoId/videos/:videoId/download`
- `GET /api/v1/repositories/:repoId/videos/:videoId/thumbnail`

## 5. API 인증 처리

인증 middleware는 성공 시 `req.auth`와 `req.user`를 함께 설정한다.

```text
req.auth = {
  kind: "dashboard" | "app" | "python",
  userId,
  role,
  displayName
}
```

모든 credential은 token payload만 신뢰하지 않고 DB에서 사용자의 활성 상태와 실제 role을 다시 확인한다.

## 6. Repository 권한 모델

### 6.1 시스템 역할

| role | 의미 |
| --- | --- |
| `admin` | 전체 시스템 관리자 |
| `user` | 일반 사용자 |

### 6.2 Repository 역할

| role | 의미 |
| --- | --- |
| `read` | repository 및 video 조회/재생 가능 |
| `maintain` | stream 등록, recording stop, video 삭제 등 운영 권한 |
| `admin` | repository metadata 수정, 멤버 관리, repository 삭제 가능 |

### 6.3 Visibility

| visibility | 의미 |
| --- | --- |
| `public` | membership이 없어도 `read` 접근 가능 |
| `private` | member 또는 system admin만 접근 가능 |

repository 접근 권한은 `repositoryService.getRepositoryAccess()`가 해석한다. system admin은 `admin`, membership은 해당 role, public repository는 membership 없이 `read`로 해석된다.

## 7. 파일 접근 제어

generated file은 직접 공개되지 않는다.

- dashboard processed playback: video detail 응답의 signed `dashboard_video_url`
- dashboard/Python download: repo-scoped `download` endpoint 통과 후 signed `/files/*` URL로 `307` redirect
- dashboard/Python thumbnail: repo-scoped thumbnail endpoint
- Python metadata: repo-scoped manifest endpoint

`/files/*`는 bearer/cookie 인증이 아니라 `signature` query string의 JWT 서명을 검증한다. 서명된 path와 실제 요청 path가 정확히 일치해야 하며, path segment validation으로 traversal을 차단한다.

## 8. MediaMTX 인증

MediaMTX는 `authMethod: http`로 동작하지만, 현재 backend `/api/v1/auth/rtmp`는 publish 인증만 담당한다.
`mediamtx.yml`의 `authHTTPExclude`가 `read`, `playback` action을 제외하므로 HLS 재생 시에는 이 callback이 호출되지 않는다.

### 8.1 publish 인증

publish는 app JWT가 아니라 publish-ticket으로 인증한다.

1. app이 app JWT로 `POST /streams/:recordingSessionId/publish-ticket` 호출
2. backend가 Redis에 publish ticket과 owner claim 생성
3. app이 `rtmp://.../live/{repo}?ticket={opaque_ticket}`로 publish
4. MediaMTX가 `/api/v1/auth/rtmp` 호출
5. backend가 ticket과 current owner lease를 검증

publish auth는 owner truth를 mutate하지 않는다. `RecordingSession`의 `PENDING -> STREAMING` 전환은 `stream-ready` hook이 담당한다.

### 8.2 read / playback 인증

Dashboard live page는 cookie session으로 `GET /api/v1/live-streams`를 호출해 접근 가능한 active stream 목록과 각 stream의 `hls_path`를 받는다.

이후 HLS 재생 흐름은 아래와 같다.

1. 브라우저가 `/hls/live/{repo}/index.m3u8`를 요청한다.
2. Caddy가 `forward_auth`로 backend `GET /api/v1/hls-auth?path=...`를 호출한다.
3. backend `requireDashboardOrAppOrPython`가 dashboard cookie, app JWT, python token(`ef_`) 중 하나를 검증한다.
4. `hls-auth.service`가 활성 live session 존재 여부와 repository read 권한을 확인한다.
5. 인증은 없거나 invalid이면 401, stream 없음 또는 repo read 권한 부족이면 404를 반환한다.
6. Caddy는 200일 때만 MediaMTX `:8888`으로 proxy한다.

즉 현재 playback에는 별도 `efp_` token이 없고, 기존 인증 수단과 repository 권한 모델을 그대로 사용한다.

## 9. 관리자 및 셀프 서비스 권한

관리자 API는 dashboard session과 system `admin` role이 필요하다.

- `/api/v1/admin/users`
- `/api/v1/admin/settings`
- `/api/v1/admin/api-tokens`
- `/api/v1/admin/users/:userId/delete-readiness`
- `/api/v1/admin/users/:userId/permanent`

사용자 본인 기능도 dashboard session이 필요하다.

- `PUT /api/v1/users/me/password`
- `POST /api/v1/auth/tokens`
- `GET /api/v1/auth/tokens`
- `DELETE /api/v1/auth/tokens/:tokenId`

admin은 본인 token은 같은 endpoint로 관리하며, admin 권한이면 타 사용자 token revoke도 가능하다.

## 10. 구현상 주의할 점

- seed는 admin 계정이 없을 때만 생성하며, 기존 admin 비밀번호를 덮어쓰지 않는다.
- app JWT payload의 role과 DB의 실제 role이 다르면 backend는 새 token을 내려 role 변화를 반영한다.
- Python token은 refresh하지 않으며 `X-Refreshed-Token`도 내려주지 않는다.
- dashboard session token은 Redis hash-key로, Python token은 DB hash column으로 원본을 저장하지 않는다.
- 비활성 사용자(`isActive=false`)는 기존 dashboard session, app JWT, Python token이 있어도 인증에 실패한다.
- admin의 user lifecycle은 deactivate 우선이다. permanent delete는 deactivated user이면서 remaining repository ownership, membership, recording history reference가 없을 때만 허용된다.
- stream stop이나 unexpected disconnect 후 live pointer가 제거되면 MediaMTX read/playback auth도 더 이상 통과하지 못한다.
