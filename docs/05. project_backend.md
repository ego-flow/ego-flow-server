# EgoFlow Server Backend

이 문서는 현재 `ego-flow-server/backend` 패키지의 구조와 책임을 정리한 문서다.

## 1. 역할

backend는 아래 책임을 가진다.

- REST API 제공
- dashboard cookie session / app JWT / Python token 인증 및 repository 권한 검사
- MediaMTX RTMP/HLS 인증 처리
- MediaMTX ready/not-ready/segment hook 처리
- generated file 접근 제어
- Prisma/Redis/BullMQ 연결 관리

## 5. 요청 처리 흐름

대부분의 요청은 다음 패턴을 따른다.

```text
route
→ auth / role / repo-access / validation middleware
→ service
→ prisma / redis / storage / queue
→ response
```

## 6. backend가 직접 관리하는 외부 시스템

- PostgreSQL
- Redis
- BullMQ
- MediaMTX auth/webhook/API 연동
- 로컬 파일 시스템의 generated dataset 경로

## 7. 구현상 특징

- 모든 주요 입력은 Zod schema로 검증한다.
- repository 권한 해석은 `repository.service.ts`에 집중되어 있다.
- dashboard session은 Redis에 SHA-256 hash 기반 key로 저장하고, cookie에는 opaque session token만 보관한다.
- Python token은 `api_tokens` 테이블에 SHA-256 hash만 저장하고, 원본은 발급 직후 한 번만 반환한다.
- `/api/v1/info`는 인증 없이 package bootstrap에 필요한 capability와 base URL 정보를 제공한다.
- video API는 flat `/api/v1/videos/*`를 제거하고 `/api/v1/repositories/:repoId/videos/*` 하나로 통합되었다.
- `GET /api/v1/repositories/:repoId/manifest`는 Python package가 dataset 전체 metadata를 페이지 단위로 읽는 canonical manifest endpoint다.
- repository video detail은 dashboard playback을 위해 만료 시간이 있는 signed `dashboard_video_url`을 포함한다.
- manifest는 각 completed video의 canonical `download_url`, `size_bytes`, `sha256`를 함께 반환하고, package는 다운로드 완료 후 이 값을 기준으로 무결성을 검증한다.
- repo-scoped `download` endpoint는 dashboard cookie session 또는 Python token 인증 뒤 signed `/files/*` URL로 `307` redirect 하고, dashboard download와 Python package download가 이 entrypoint를 공유한다.
- 따라서 package의 dataset access 흐름은 `manifest 조회 -> download endpoint 호출 -> signed file redirect -> 로컬 size/hash 검증` 순서로 구성된다.
- repo-scoped `thumbnail` endpoint는 bearer auth 뒤 이미지를 직접 streaming한다.
- `/files/*`도 일반 정적 파일 서빙이 아니라 signed URL 검증을 거친다.
- worker는 별도 패키지가 아니라 backend 패키지 내부 엔트리로 분리되어 있다.
