# EgoFlow Server Frontend

이 문서는 현재 `ego-flow-server/frontend` 패키지의 구조와 역할을 정리한 문서다.

## 1. 역할

frontend는 dashboard UI를 제공한다.

- 로그인 및 세션 유지
- repository 목록/상세/설정 화면
- video 목록/상세/재생 화면
- live stream 모니터링
- 관리자 사용자 및 Python token 관리
- 본인 비밀번호 변경 및 Python token self-service

## 3. 주요 구성 요소

### 3.1 routes

TanStack Router의 file-based route 구조를 사용한다.

| 경로 | 역할 |
| --- | --- |
| `/login` | 로그인 |
| `/repositories` | repository 목록 |
| `/repositories/new` | repository 생성 |
| `/repositories/$repoId` | repository 상세, video filter/sort/pagination |
| `/repositories/$repoId/settings` | repository 설정 및 member 관리 |
| `/repositories/$repoId/videos/$videoId` | video 상세 및 재생 |
| `/live` | active stream 모니터 |
| `/admin/users` | 사용자 관리, active Python token revoke |
| `/admin/settings` | 서버 설정 조회 |
| `/profile` | 본인 비밀번호 변경, Python token 발급/회전/폐기 |

### 3.2 api

backend 응답 형식을 frontend 친화적 모델로 변환하는 계층이다.

| 파일 | 역할 |
| --- | --- |
| `client.ts` | axios client, auth header, token refresh 처리 |
| `repositories.ts` | repository API adapter |
| `videos.ts` | repo-scoped video API adapter |
| `streams.ts` | active stream API adapter |
| `admin.ts` | admin API adapter |
| `tokens.ts` | Python token 발급/조회/revoke adapter |

### 3.3 hooks / lib

| 파일 | 역할 |

| `hooks/useAuth.tsx` | auth context |

### 3.4 components

## 4. 인증 처리 방식

- 로그인 성공 시 backend가 HttpOnly `egoflow_session` cookie를 설정하고, frontend는 user 정보만 React state에 보관한다
- page refresh 시 `/api/v1/auth/dashboard/session`으로 cookie session을 확인한다
- axios는 `withCredentials: true`로 same-origin 또는 허용된 origin에 cookie를 전송한다
- Python token은 dashboard 세션 저장소에 보관하지 않고, `/profile`에서 발급 직후 한 번만 노출한다
- thumbnail 이미지는 cookie credential을 포함한 `fetch()`로 받아 `blob:` URL로 렌더링한다
- processed video playback은 backend가 내려준 signed URL을 직접 사용한다
- processed video download는 video 상세 화면의 Download 버튼이 repo-scoped `download` endpoint를 cookie session으로 호출한 뒤, redirect된 signed file 응답을 `blob`으로 저장한다

## 5. 재생 구조

### 5.1 live 재생

- backend가 반환한 origin-relative HLS path(`/hls/...`)를 frontend가 그대로 사용하고, 브라우저가 현재 dashboard origin 기준으로 요청한다
- HLS absolute base URL을 별도 env로 관리하지 않는다
- live 목록 응답의 `hls_path`를 `HlsPlayer`에 그대로 전달하고, 브라우저가 현재 dashboard origin 기준으로 `/hls/...`를 요청한다
- `HlsPlayer`는 `hls.js` XHR에 `withCredentials = true`를 설정해 dashboard session cookie가 Caddy `forward_auth`까지 전달되게 한다

### 5.2 processed video 재생

- metadata 조회는 `/api/v1/repositories/:repoId/videos/:videoId`
- thumbnail 조회는 repo-scoped thumbnail endpoint를 cookie credential 포함 `fetch()`로 읽고 `blob:` URL로 렌더링한다
- dashboard playback은 detail 응답의 signed `dashboard_video_url`을 그대로 사용한다
- dashboard download는 video 상세 화면의 Download 버튼에서 repo-scoped `download` endpoint를 cookie session으로 호출하고, backend redirect 뒤 최종 signed `/files/*` 응답을 `blob`으로 저장한다

## 6. 주요 화면 흐름

### 6.1 Repository 중심 탐색

사용자는 `/repositories`에서 접근 가능한 repository를 보고, 각 repository 상세에서 repository 내부 video 목록을 필터링/정렬/페이지 이동하며 탐색한다.

### 6.2 Live 모니터

`/live`는 active stream 목록을 polling하며, 선택한 repository의 live HLS를 재생한다.

### 6.3 Admin 화면

- `/admin/users`: 사용자 생성, 비밀번호 재설정, 비활성화, active Python token revoke
- `/admin/users`의 Create user form은 form submit 한 번만 전송하며, 이미 존재하는 user id는 backend `409 Conflict`로 표시한다
- `/admin/settings`: 현재 target directory 조회

### 6.4 Profile 화면

- 비밀번호 변경
- Python token 이름 입력 후 발급 또는 회전
- 현재 active token 메타데이터 확인
- 발급 직후 raw token 모달 표시와 복사
- token revoke

## 7. 구현상 특징

- backend snake_case 응답을 camelCase 모델로 변환한다.
- UI는 repository 기준 내비게이션으로 구성되어 있다.
- video API 호출은 모두 repo-scoped backend endpoint로 통일되어 있다.
- video 상세 화면은 processing 상태일 때 polling으로 진행률을 갱신한다.
